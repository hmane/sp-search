## Part 1 — The Two Journeys

### Journey A: Day 1 Admin Install

The 12 steps below trace what an unaccompanied tenant admin does, in order, from `.sppkg` in hand to working search experience. Friction is logged inline with severity ([Blocker] / [Confusion] / [Polish] per spec §4.2) and an owning-track pointer (T1–T5 or Foundations).

#### Step 1 — Download / receive `.sppkg`

The admin obtains the packaged `.sppkg`. There is no published release artifact in the repo, no GitHub Releases entry, and no pinned download link in `docs/deployment-guide.md`. The documented path to produce the artifact yourself is `npm install && npm run type-check && npm test -- --runInBand && gulp bundle --ship && gulp package-solution --ship` (`docs/deployment-guide.md:17-23`).

**Friction:**
- **[Blocker]** No `gulpfile.js` exists in the repo (verified at HEAD), but `docs/deployment-guide.md:21-22` and `scripts/Setup-SPSearchSite.ps1:7,135` both prescribe `gulp bundle --ship && gulp package-solution --ship`. SPFx 1.22 / Heft migration replaced gulp with `heft build` / `heft package-solution` (per `package.json:21,22` `"package": "heft build --clean --production && heft package-solution --production"`), but no doc was updated. An admin following the docs verbatim will hit "command not found: gulp". → owning track: Foundations
- **[Blocker]** `npm run type-check` resolves to `heft build --clean --lite` (`package.json:28`), but `--lite` is not a documented Heft CLI flag in current Heft releases. The script either fails or produces no useful output. (See Foundations finding F-2.) → owning track: Foundations
- **[Blocker]** `npm test` (which the deployment guide tells the admin to run before packaging at `docs/deployment-guide.md:20`) fails because `src/styles/pnpPropertyControlsFix.ts:33` references the constant `PNP_COLLECTION_DATA_CSS` before its declaration on `:42`, tripping a `no-use-before-define` lint error that Heft elevates to a build failure. (See Foundations finding F-1.) `npm run package` is similarly blocked. → owning track: Foundations
- **[Confusion]** `docs/deployment-guide.md:164` references "the current `gulpfile.js`" and `npm run serve` — neither exists in the repo (no `serve` script in `package.json:17-33`; `start` exists). → owning track: Foundations
- **[Polish]** No `webApiPermissionRequests` declared in `config/package-solution.json` (verified — array absent), so the Graph People vertical (which `docs/admin-guide.md:240-242` says requires `People.Read`) silently no-ops on Day 1 because the API access page in SharePoint admin center has nothing to approve. The admin must know to add the permission request manually. → owning track: T4 (Admin Experience) / Foundations

#### Step 2 — Upload to tenant or site app catalog

With a built `.sppkg` in hand, the admin uploads to a tenant or site app catalog. `docs/deployment-guide.md:33-43` documents both paths in two short sub-sections; `scripts/Setup-SPSearchSite.ps1:142-208` automates the site-collection app catalog path end-to-end (enables catalog, polls for readiness, uploads via REST to bypass the PnP 200-second `Add-PnPApp` timeout, deploys, installs).

**Friction:**
- **[Polish]** Tenant app catalog upload has no automation script in `scripts/`. Only the site-collection catalog path is automated (`Setup-SPSearchSite.ps1:142-208`). Multi-site rollouts must use ad-hoc PowerShell or the SharePoint admin UI. → owning track: T4
- **[Polish]** `solution.developer.mpnId` is `"Undefined-1.21.1"` (`config/package-solution.json:15`) — a generator default that displays in the admin "Apps you can add" pane. It signals "experimental scaffold" rather than "shipping product" to admins reviewing third-party packages. → owning track: T4

#### Step 3 — Add app to a site

After deployment, the admin opens the target site's `Site contents` and adds the SP Search app. With `skipFeatureDeployment: true` (`config/package-solution.json:8`) the web parts become available tenant-wide once deployed, so the explicit add-app step is effectively a no-op for tenant-catalog deployments — but `docs/deployment-guide.md:47` still instructs the admin to do it.

**Friction:**
- **[Confusion]** `docs/deployment-guide.md:47` says "After deployment, add the SP Search app to the target site from **Site contents**" without explaining that `skipFeatureDeployment: true` (`config/package-solution.json:8`) makes the add-app step optional for tenant-catalog deployments and required for site-catalog deployments. The admin who skips it on a site-catalog deploy will see no web parts in the toolbox and not understand why. → owning track: Foundations (docs)

#### Step 4 — Run provisioning script (`Setup-SPSearchSite.ps1`)

The admin runs `scripts/Setup-SPSearchSite.ps1 -SiteUrl ... -ClientId ...` — a 557-line one-shot orchestrator that connects, deploys the `.sppkg` (Phase 2), provisions the three hidden lists `SearchSavedQueries` / `SearchHistory` / `SearchCollections` (Phase 3, lines 253-364), creates a Search page with five web parts wired together (Phase 4, lines 369-492), and publishes it (Phase 5, line 499). Required parameters are `-SiteUrl` and `-ClientId` (mandatory at `:23-27`).

**Friction:**
- **[Blocker]** `-ClientId` is `Mandatory = $true` at `Setup-SPSearchSite.ps1:26-27` with no fallback to the environment-variable convention (`ENTRAID_APP_ID` / `ENTRAID_CLIENT_ID` / `AZURE_CLIENT_ID`) used by the standalone scripts (e.g. `Provision-SPSearchLists.ps1:39-62`, `Search-ScenarioPresets.ps1:393-411`). An admin who has not pre-registered an Entra app cannot run the one-shot script at all — and the prerequisites table in `docs/deployment-guide.md:7-13` does not mention the Entra app registration requirement. → owning track: T4
- **[Blocker]** `Setup-SPSearchSite.ps1:373-377` unconditionally **deletes any existing page** named `Search` (the default `-PageName` value at `:33`) via `Remove-PnPPage -Identity $PageName -Force` before recreating it. There is no confirmation prompt, no `-WhatIf` support on the script as a whole (only `Search-ScenarioPresets.ps1:528` adds `[CmdletBinding(SupportsShouldProcess)]`). A second run silently destroys page customisations. → owning track: T4
- **[Confusion]** `Setup-SPSearchSite.ps1:488` calls `Get-SeededCoverageProfiles -BaseSiteUrl $SiteUrl` which returns coverage profiles pointing at 10 hardcoded document libraries (`CorporatePolicies`, `SalesMaterials`, …, lines 50-61) plus 10 hardcoded custom lists (`Projects`, `Contacts`, …, lines 63-74). On any tenant that has not separately run `scripts/Provision-TestData.ps1`, every coverage profile points at non-existent URLs and the Admin Search Manager's Coverage tab opens to broken stat cards on first view. → owning track: T4
- **[Confusion]** `Setup-SPSearchSite.ps1:243` does `Start-Sleep -Seconds 5` after install with the comment "Allow app to register", but the next phase (list provisioning) does not depend on web part registration — only Phase 4 does. The fixed sleep adds latency without addressing the actual race documented elsewhere in the codebase (e.g. `Add-PnPApp` / `Get-PnPTerm` timing notes in MEMORY.md). → owning track: Foundations
- **[Polish]** `Setup-SPSearchSite.ps1:46-48` defines web part component names as string literals: `WP_SEARCH_BOX = "SP Search Box"`, `WP_SEARCH_FILTERS = "SPSearchFilters"`, `WP_SEARCH_VERTICALS = "SPSearchVerticals"`. The Filters and Verticals strings differ from the manifest `preconfiguredEntries[].title.default` (`SP Search Filters` at `SpSearchFiltersWebPart.manifest.json:21`, `SP Search Verticals` at `SpSearchVerticalsWebPart.manifest.json:21`). PnP `Add-PnPPageWebPart -Component` matches by display name — these will silently insert blank web parts in some PnP versions, per the `MEMORY.md` "PnP.PowerShell 3.x Gotchas" note. → owning track: Foundations

#### Step 5 — Run scenario presets script (`Search-ScenarioPresets.ps1`)

Optional. The admin dot-sources `scripts/Search-ScenarioPresets.ps1` and calls `Invoke-SearchScenarioPage -SiteUrl ... -PageName ... -ScenarioName ...` to create a fully-configured page for a built-in scenario (`general`, `documents`, `people`, `news`, `media`, `hub-search`, `knowledge-base`, `policy-search`, `account-documents` — verified at `Search-ScenarioPresets.ps1:39-358` and the TS registry at `src/webparts/spSearchResults/presets/searchPresets.ts:424-434`).

**Friction:**
- **[Confusion]** Both `Setup-SPSearchSite.ps1` (Step 4) and `Invoke-SearchScenarioPage` (this step) create a search page with the same five web parts but with **different defaults**: Setup uses `-PageName "Search"` and provisions Verticals + Filters explicitly with hardcoded JSON (`Setup-SPSearchSite.ps1:389-411`); Invoke uses preset-driven configuration (`Search-ScenarioPresets.ps1:587-657`). Running both on the same site produces two pages with subtly different filter sets, sort options, and page sizes (Setup uses `pageSize=25` at `:459`; Invoke uses `pageSize=10` at `:599`). The admin has no signal that "you should pick one of these, not both." → owning track: T4
- **[Confusion]** `Invoke-SearchScenarioPage:629-634` writes `filterCollection` and `Search-ScenarioPresets.ps1:653-657` writes `verticalCollection` as the property name, but `SpSearchFiltersWebPart.manifest.json:30` and `SpSearchVerticalsWebPart.manifest.json:27` use `filtersCollection` (plural) and `verticalsCollection` (plural) respectively. Filters and Verticals will fall back to manifest defaults instead of the preset's curated set, silently. → owning track: T4
- **[Confusion]** `docs/deployment-guide.md:79-87` lists the 8 built-in presets but `docs/admin-guide.md:111-120` lists 9 (adds `custom`); neither doc lists `account-documents` (`searchPresets.ts:333,433`). → owning track: Foundations (docs)
- **[Polish]** `Search-ScenarioPresets.ps1` is not a runnable script — it is a function library that requires the admin to know to dot-source it (`. .\scripts\Search-ScenarioPresets.ps1`) before calling `Invoke-SearchScenarioPage`. This convention is not documented in `docs/deployment-guide.md:71-87`. → owning track: Foundations (docs)

#### Step 6 — Open a page in edit mode, add Search Box / Verticals / Filters / Results / Manager

If the admin chose not to use Steps 4 or 5's automated page creation, they edit a SharePoint page and add five web parts from the toolbox. Each ships with an icon (`SpSearchBoxWebPart.manifest.json:23` `Search`, `SpSearchResultsWebPart.manifest.json:23` `Search`, `SpSearchFiltersWebPart.manifest.json:23` `Filter`, `SpSearchVerticalsWebPart.manifest.json:23` `SortLines`, `SpSearchManagerWebPart.manifest.json:23` `SearchAndApps`) and a one-line description (`*.manifest.json:22`).

**Friction:**
- **[Confusion]** Two admin-manager web parts ship in the same package: `SpSearchManagerWebPart` titled "SP Search Manager (Legacy)" with description "Legacy admin manager entry. Use SP Search Admin Manager for new admin pages." (`SpSearchManagerWebPart.manifest.json:21-22`), and `SpSearchAdminManagerWebPart` titled "SP Search Admin Manager" (`SpSearchAdminManagerWebPart.manifest.json:14`). Both share the icon `SearchAndApps`. A first-time admin sees two near-identical entries in the toolbox with no other differentiation. (See INC-007 in Appendix A — Changed-Form: behavior is 100% inherited; only manifest defaults differ plus the DebugCollector hook in `onInit`.) → owning track: T4
- **[Confusion]** Three of the five web parts use `Search` or `SearchAndApps` as their Fluent icon (`SpSearchBoxWebPart.manifest.json:23`, `SpSearchResultsWebPart.manifest.json:23`, `SpSearchManagerWebPart.manifest.json:23`, `SpSearchAdminManagerWebPart.manifest.json:16`). In the SPFx toolbox preview, four of six tiles look identical at a glance. → owning track: T1
- **[Polish]** Web part icons are Fluent v8 icon-font glyphs (`officeFabricIconFontName`), not custom SVG/PNGs — so the toolbox preview is always monochrome 16×16. PnP Modern Search ships with custom illustrative icons that read as branded entries. → owning track: T1
- **[Polish]** `SpSearchBox` defaults `enableSearchManager: true` (`SpSearchBoxWebPart.manifest.json:41`), which adds a Search Manager button inside the search box. If the admin also adds a separate Search Manager web part to the page, the user sees two manager entry points. → owning track: T2

#### Step 7 — Configure searchContextId across web parts

The admin opens each of the five web parts' property panes and verifies the `searchContextId` matches. The default value is `default` on every manifest (`SpSearchBoxWebPart.manifest.json:25`, `SpSearchResultsWebPart.manifest.json:25`, `SpSearchFiltersWebPart.manifest.json:25`, `SpSearchVerticalsWebPart.manifest.json:25`, `SpSearchManagerWebPart.manifest.json:25`, `SpSearchAdminManagerWebPart.manifest.json:18`), so a single search experience per page works without action. For multi-context pages or when the admin wants meaningful IDs (e.g. `hr-search`), they must edit each web part.

**Friction:**
- **[Blocker]** `searchContextId` is buried at the bottom of the property pane in every web part: page 3 (Connections) of 4 in Search Box (`SpSearchBoxWebPart.ts:362-375`); page 3 (Connections) of 4 in Search Results (`SpSearchResultsWebPart.ts:1207-1228`); page 2 (Connections) of 3 in Search Filters (`SpSearchFiltersWebPart.ts:443-463`); page 1 group 1 of 2 in Verticals (`SpSearchVerticalsWebPart.ts:317-330`, the only one with it on top); page 1 group 1 in Manager (`SpSearchManagerWebPart.ts:201-215`). For a setting that **must** match across five web parts, the admin has to navigate to a different pane location in each. → owning track: T3 (Multi-Instance / Multi-Context)
- **[Confusion]** When two web parts have **different** IDs by accident, there is no edit-mode warning anywhere. `SpSearchResults.tsx:716-720` only emits an info bar when `searchContextId === 'default'`, advising to set a unique ID for multi-context pages. There is no detection of "Box uses `default`, Results uses `hr-search`, no shared store". → owning track: T3
- **[Confusion]** Required-field error messages diverge across web parts. Search Box says "Required — must match the Search Context ID set on the Search Results web part." (`SpSearchBoxWebPart.ts:369`); Search Results says "Required — enter an ID to connect search web parts on this page (e.g. \"hr-search\")." (`SpSearchResultsWebPart.ts:1218-1220`); Filters has no `onGetErrorMessage` (`SpSearchFiltersWebPart.ts:450-462`); Verticals/Manager likewise lack validation. → owning track: T3
- **[Polish]** Description text differs across web parts: Box says "Web parts with the same context ID share search state. Leave blank for 'default'." (`spSearchBox/loc/en-us.js:17`); Results says "Identifier to connect this web part with other search web parts (search box, filters, verticals)." (`spSearchResults/loc/en-us.js:17`); Filters says "Links this filter web part to a search results web part that shares the same context ID" (`spSearchFilters/loc/en-us.js:14`). Same setting, three different mental models. → owning track: T3

#### Step 8 — Open property panes, configure scope / filters / columns / layout

The admin tunes the Results scope and query (`SpSearchResultsWebPart.ts:944-1000`, page 1), columns (`:976-999`, `:1129-1145`, `:1151-1167`), layout preset and toggles (`:1078-1124`, page 2), filters (`SpSearchFiltersWebPart.ts:255-415`, page 1), and verticals (`SpSearchVerticalsWebPart.ts:331-356`).

**Friction:**
- **[Confusion]** The Search Results property pane has 4 pages with 11 collapsible groups (`SpSearchResultsWebPart.ts:946,1003,1042,1076,1107,1127,1149,1171,1212,1235,1250` — `DataGroupName`, `SortGroupName`, `PaginationGroupName`, `MainLayoutsGroupName`, `AdvancedLayoutsGroupName`, `CompactViewGroupName`, `GridViewGroupName`, `BehaviorGroupName`, `ConnectionsGroupName`, `QuerySettingsGroupName`, `AdvancedGroupName`). The scenario preset picker (`PropertyPaneChoiceGroup('layoutPreset')` at `:1078`) is on **page 2** ("Display") inside the `MainLayoutsGroupName`, not on page 1 — admins who never reach page 2 miss the curated starter that would set query template, columns, sort, and filters atomically. → owning track: T4
- **[Confusion]** Changing the scenario preset only updates the Results web part; it does **not** push the matching `filterSuggestions` or `verticalSuggestions` from `searchPresets.ts:60-66, 92-104, 130-137` into the Filters and Verticals web parts. The admin must manually re-key those settings on the other web parts to get the preset's intended behaviour. → owning track: T4
- **[Confusion]** Edit-mode validation in `SpSearchResults.tsx:722-750` warns about default-layout-not-enabled, grid-with-no-columns, and similar property-level misconfigurations, but there is **no** validation that the chosen managed property names (`selectedPropertiesCollection.property`, `refinementFiltersCollection.property`, `sortablePropertiesCollection.property`) actually exist in the tenant search schema. `PropertyPaneSchemaHelper` only assists for `queryTemplate` (`SpSearchResultsWebPart.ts:965-971`); column pickers fall back to free-text strings (`:984-996`). A typo like `LastModifedTime` results in a silently empty column at runtime. → owning track: T4
- **[Confusion]** `SpSearchResultsWebPart.ts:268-270` resets `layoutPreset` to `'custom'` if it isn't set, and `:888-890` reverts to `'custom'` whenever the admin toggles any layout-related property. The admin who picks `documents`, then enables `showCardLayout` to add Card view, silently demotes the preset to `custom` — and any subsequent prop changes will no longer track the `documents` defaults. → owning track: T4
- **[Polish]** Coverage Profiles in the Admin Manager pane (`SpSearchManagerWebPart.ts:266-345`) is a 10-column `PropertyFieldCollectionData` that requires the admin to type comma-separated URLs, content type IDs (e.g. `0x0101`), and refinement filters (`FileType:or("docx","pdf")`) into free-text cells. No autocomplete, no validation, no schema picker. → owning track: T4

#### Step 9 — Run a test query

In edit mode the admin types a query in the Search Box. The orchestrator triggers a search, and Results renders matching rows or one of four `EmptyState` messages (`SpSearchResults.tsx:402-421`).

**Friction:**
- **[Confusion]** Until the admin has run `scripts/Map-CrawledProperties.ps1` (`Map-CrawledProperties.ps1:1-68`) and waited for a re-index, custom managed properties referenced in starter filters or columns return zero buckets. The deployment guide does not mention this dependency in its "Smoke Test Checklist" (`docs/deployment-guide.md:140-153`). → owning track: T4
- **[Confusion]** Map-CrawledProperties hardcodes 13 specific `ows_SPS*` crawled properties (`Map-CrawledProperties.ps1:54-68`) — properties that only exist on tenants that have provisioned the test-data site columns. There is no in-product guidance for an admin whose tenant has different custom columns (e.g. `ows_Contoso_*`). → owning track: T4
- **[Polish]** When `searchContextId === 'default'` the Results web part shows an info `MessageBar` in edit mode ("Using the default search context...", `SpSearchResults.tsx:716-720`) but does not render in view mode. End users on a published page won't see the warning. The signal that a developer-default ID is still in use disappears at exactly the moment it would matter for downstream multi-page collisions. → owning track: T3

#### Step 10 — Configure saved searches, sharing, history retention

The admin enables Search Manager features: saved searches, shared searches, history, collections. The Manager web part exposes `enableSavedSearches` / `enableSharedSearches` / `enableCollections` / `enableHistory` toggles plus `maxHistoryItems` (`SpSearchManagerWebPart.manifest.json:30-40`). History retention is not configured here.

**Friction:**
- **[Confusion]** **Documented defaults disagree with shipped defaults.** `docs/admin-guide.md:221-226` says `enableSavedSearches=true`, `enableSharedSearches=true`, `enableCollections=true`, `enableHistory=true`. Both manifests ship with all four set to `false` (`SpSearchManagerWebPart.manifest.json:30-33`, `SpSearchAdminManagerWebPart.manifest.json:25-26,29`). An admin who reads the docs and adds the web part will see an empty manager with every tab disabled. → owning track: Foundations (docs)
- **[Confusion]** History retention runs automatically but is invisible and unconfigurable to the admin. `SearchManagerService.ts:735-739` enforces a 24-hour `HISTORY_CLEANUP_INTERVAL_MS` that calls `cleanupHistory(HISTORY_RETENTION_DAYS)` (90-day default, defined at `:19-20`; cleanup implementation at `:1417`), so the `SearchHistory` list is bounded by the daily sweep at the row level and by the per-item `MAX_CLICKED_ITEMS = 10` ceiling (`SearchManagerService.ts:857-860`, see INC-008 Closed in Appendix A) at the field level. The friction is twofold: (1) `docs/provisioning-guide.md:131-132` is stale — it states "This is a manual operation — there is no automatic background cleanup," directly contradicting the shipped code, so an admin reading the docs will be misinformed about how (or whether) retention works; and (2) `grep cleanupHistory src/webparts/` returns zero hits — there is no property pane field, dashboard tab, status badge, or trigger button anywhere in the published web parts, so the admin cannot view the retention window, override the 90-day default, or force an on-demand sweep without reading source. → owning track: Foundations (stale docs) with secondary T4 (admin lacks visibility / override)
- **[Confusion]** Item-level permission behaviour is documented for `SearchHistory` (`Provision-SPSearchLists.ps1:230-234` sets `ReadSecurity=2 WriteSecurity=2`) but the admin has no in-product confirmation that this was applied. If the lists were created manually or by a partial script run, the admin can leak other users' history. → owning track: T4
- **[Polish]** Saved-search JSON is parsed without schema validation (`SavedSearchList.tsx:76,163`, see SEC-004 Still-Open in Appendix A). An admin sharing a tampered saved search to colleagues can poison their stores. → owning track: Foundations

#### Step 11 — Publish the page

The admin clicks Publish in the SharePoint editor. Both `Setup-SPSearchSite.ps1:499` and `Search-ScenarioPresets.ps1:706` automate this when the scripts are used; manual edits require the admin to publish themselves.

**Friction:**
- **[Polish]** The Admin Search Manager's Coverage tab pulls live coverage stats on first view (`AdminDashboard.tsx:88-90`). If the seeded coverage profiles still point at the test-data hardcoded URLs (see Step 4 friction), the published page renders a "broken stats" panel as the very first thing the admin or end user sees post-publish. → owning track: T4
- **[Polish]** No published-page smoke check is automated. The "Smoke Test Checklist" at `docs/deployment-guide.md:140-153` is a manual 8-row table (Type a query, Switch verticals, Apply author filter, Switch to Grid, Export CSV/XLSX, Open Health, Open Insights, Open a People result). On a fresh tenant with no crawled content, six of the eight will produce empty output and the admin has no way to distinguish "feature works, no data" from "feature broken". → owning track: T5 (Observable & Diagnosable)

#### Step 12 — Hand off to end users

The admin publishes the URL and (typically) sends an email or Teams message to end users. There is no in-product onboarding tour, in-page help link, or admin-configurable banner.

**Friction:**
- **[Confusion]** No end-user documentation ships in `docs/`. The available docs are admin-facing (`docs/admin-guide.md`, `docs/deployment-guide.md`, `docs/provisioning-guide.md`, `docs/extensibility-guide.md`, `docs/pnp-modern-search-alignment.md`). End users have nothing to read about saved searches, KQL mode, scope selector, layout switching, or sharing. → owning track: T2 / Foundations (docs)
- **[Polish]** The branch carrying the current SPFx 1.22 / Heft migration (`feat/spfx-1.22-heft-migration`) is unmerged at handoff time (74 commits ahead of `main` per `git log --oneline main..HEAD | wc -l`). Admins who clone and build from `main` will get the prior SPFx 1.21 build chain; admins who clone the feature branch get the un-shipped Heft path. There is no published versioning policy or release tag to point them at. (Foundations finding — the unmerged-branch concern is captured for Foundations Track per spec §4.4 "SPFx 1.22 / Heft migration completion".) → owning track: Foundations
- **[Polish]** No admin telemetry surface. The product has a Debug FAB and Admin Dashboard (`AdminDashboard.tsx:1-280`), but no "send a support bundle" / "export current state" path that an admin can give a user to attach to a ticket. (Per spec §4.3 T5: "exportable support bundle".) → owning track: T5

### Journey B: Day 1 End-User Search

The 12 steps below trace what an unaccompanied end user does the first time they hit a published SP Search page — they have no docs, no source, no admin to ask. Friction is logged inline with severity ([Blocker] / [Confusion] / [Polish] per spec §4.2) and an owning-track pointer (T1–T5 or Foundations). Admin-side issues already logged in Journey A are not duplicated here.

#### Step 1 — Lands on a search page (no prior context)

The user opens the page and sees five web parts wired into the layout: a Search Box at the top, Verticals tabs, Filters in a sidebar, Results in the main area, and (optionally) a Search Manager. With no query yet, `SpSearchResults.tsx:598-600` short-circuits `renderLayout()` to `<LoadingShimmer count={5} />` because `hasSearched === false`. The Filters web part renders its own message — "No filters available. Perform a search to see available filters." (`SpSearchFilters.tsx:409-411`). Verticals render with no count badges yet. The Search Box placeholder is `"Search this site"` (`SpSearchBoxWebPart.manifest.json:26`).

**Friction:**
- **[Confusion]** First paint shows five list-shaped shimmer cards (`SpSearchResults.tsx:223-298, 599`) even though no search has run — the system is idle waiting for input. The idle copy "Search" / "Enter a search term to get started." (`:418-421`) only appears after `hasSearched` flips, which never happens until the user types. → owning track: T1
- **[Confusion]** No onboarding hint, no example queries, no keyboard-shortcut hint, no "Try…" suggestions for the empty page. The Search Box placeholder is the only signal. `grep -in 'help\|tooltip\|onboard\|tutorial\|welcome' src/webparts/spSearchBox/components/SpSearchBox.tsx` returns zero hits in user-facing copy. → owning track: T2
- **[Polish]** When `searchScopes` is empty (the manifest default at `SpSearchBoxWebPart.manifest.json:31`) but the admin enabled `enableScopeSelector`, `SpSearchBox.tsx:723` renders no scope dropdown at all (`scopeOptions.length > 0` guard at `:723`). The end user has no signal that "scope selection was supposed to be here." → owning track: T1
- **[Polish]** Three different loading idioms in three adjacent web parts: Filters uses three custom shimmer rows (`SpSearchFilters.tsx:419-451`); Results uses a 5-row list-shaped shimmer (`SpSearchResults.tsx:223-310`); the Search Manager panel shows a Fluent `Spinner` with "Loading search manager…" text (`SpSearchManager.tsx:672-677`). No shared visual rhythm. → owning track: T1

#### Step 2 — Types a query

The user clicks the Search Box and starts typing. `handleInputChange` (`SpSearchBox.tsx:369-373`) updates local state and debounces `requestSuggestions` by `debounceMs` (default 300, configurable). On Enter (`handleSearch` at `:380-382`) or after the debounce settles when `searchBehavior === 'onChange'`, `executeSearch` dispatches `setQueryText` to the store (`:363`), which the orchestrator picks up.

**Friction:**
- **[Confusion]** When the user types, `requestSuggestions` is gated by `value.length === 0 || value.length >= 2` (`SpSearchBox.tsx:224`). After a single character, suggestions are explicitly cleared (`:231-237`) with no UI signal — the user sees the dropdown vanish at character 1, reappear at character 2. There is no "keep typing…" placeholder or affordance. → owning track: T2
- **[Confusion]** KQL mode is disabled by default (`enableKqlMode: false`, `SpSearchBoxWebPart.manifest.json:40`). When the admin does enable it, the only entry is two unlabelled toggle pills "Search / KQL" inside the input bar (`SpSearchBox.tsx:735-762`). End users with no docs have no idea what KQL is, what syntax is accepted, or what `Author:John AND FileType:docx` means — the only hint is the `title="KQL mode — type queries like Author:John AND FileType:docx"` tooltip on hover (`:752`). → owning track: T2
- **[Polish]** Enter triggers search regardless of `searchBehavior`, but the clear-button (`X` inside `SearchBox`) calls `handleClear` (`SpSearchBox.tsx:394-410`) which immediately resets `queryText` to `''` and conditionally clears all filters when `resetSearchOnClear === true`. There is no undo for an accidental Esc/clear that wipes a user's filter set. → owning track: T2
- **[Polish]** No keyboard shortcuts to focus the search box from anywhere on the page (`/`, Ctrl+K, etc.). The user must point-and-click. → owning track: T2

#### Step 3 — Sees suggestions (or doesn't)

After two characters and a 300 ms debounce, the suggestion dropdown opens. `SuggestionDropdown.tsx:191-262` renders grouped suggestions: "SharePoint Suggestions", "People Suggestions", "Quick Results", "Recent Searches", "Popular Searches", "Search Properties". All five providers are enabled by default (`SpSearchBoxWebPart.manifest.json:33-37`).

**Friction:**
- **[Confusion]** "Recent Searches" and "Popular Searches" both query the per-user `SearchHistory` list (`RecentSearchProvider.ts:29` calls `loadHistory`, `TrendingQueryProvider.ts:74-`). On Day 1 the list is empty for this user, so both groups silently render zero items and the corresponding group headers never appear. The user sees three groups one day, five another, with no explanation of why. → owning track: T2
- **[Confusion]** Group descriptions are visible but generic — "Matches from the SharePoint search index" / "Suggested names from SharePoint search" / "Top matching items from SharePoint search" (`SuggestionDropdown.tsx:18-47`). All three describe the same backend; they read as duplicates to a non-technical user. → owning track: T1
- **[Confusion]** Suggestion ranking is provider-priority order then per-provider order (`SpSearchBox.tsx:253-256`, `mergeSuggestionsByPriority` at `:27-56`), with a deduping `seen` set on lowercased displayText. Two providers can return the same string with different secondary text (e.g., "John Smith" from People + "John Smith" from Recent), and the second one is silently dropped (`:42-44`). → owning track: T2
- **[Polish]** `Delete` removes a Recent suggestion (`SuggestionDropdown.tsx:120-125`) but no on-screen affordance announces it — only the trailing X is visible (`:245-256`). Mac users (whose Backspace key is labelled "delete") will hit the wrong key. → owning track: Foundations
- **[Polish]** Quick Results and query-text suggestions look identical in the dropdown — same row height, same icon styling — even though clicking one navigates to a document (`suggestion.action`, `SpSearchBox.tsx:439-447`) and the other re-runs a query. The difference is only in the group header. → owning track: T1

#### Step 4 — Reads the empty state if no results

The user types something obscure. The orchestrator runs the search, `items.length === 0` and `hasSearched === true`, and `EmptyState` renders (`SpSearchResults.tsx:402-444, 609-617`). Four context-aware messages cover the (queryText × hasActiveFilters) cases (per UX-002 fix, see Appendix A). Recovery offered: "Clear all filters" if filters are active, "Start over" link if either query or filters are present.

**Friction:**
- **[Confusion]** `EmptyState` shows the Fluent icon `SearchIssue` (`SpSearchResults.tsx:426`), an exclamation/glyph that reads as "warning/error" rather than "neutral no-match". A user typing a deliberately speculative term ("Q4 budget projections") will read it as "you broke something." → owning track: T1
- **[Confusion]** When there is a "Did you mean" alternative in the response, it renders **above** the empty state in a row near the top of the results pane (`SpSearchResults.tsx:767-779`), but the empty state's heading still says "No results for &quot;…&quot;" with no visual link to the suggestion. Eye-path: heading first, suggestion second — opposite of the recovery flow. → owning track: T1
- **[Polish]** Empty state offers no nudge toward Verticals — switching from "All" to "Documents" or "People" can produce results when the broad query doesn't, but the empty panel doesn't say "try another vertical" or surface vertical counts. The verticals bar above shows zero counts (when `hideEmptyVerticals === false`) or hides them (`SpSearchVerticals.tsx:122-125`), and the user can't tell which verticals are zero vs. just hidden. → owning track: T2
- **[Polish]** No spell-check / fuzzy-match hint when SharePoint Search returns no `querySuggestion`. The Empty State copy "Check your spelling or try broader search terms" (`SpSearchResults.tsx:414`) is static text, not an active aid. → owning track: T2
- **[Polish]** "Start over" reset (`SpSearchResults.tsx:567-571`) clears the URL via `window.location.href = url.toString()`, forcing a full page reload — slow, jarring, and resets all in-page state including unrelated web parts. A soft reset (`store.reset()`) would be preferable. → owning track: T1

#### Step 5 — Switches verticals

The user clicks a different vertical tab — say "Documents" — to narrow the result set. `VerticalTab.tsx:34-38` calls `setVertical(key)` which (`verticalSlice.ts:11`) sets `currentVerticalKey`, resets `currentPage: 1`, and clears `activeFilters: []`. The orchestrator triggers a fresh search with the vertical's `dataProviderId` routing (per Sprint 3 / per-vertical Graph-vs-SharePoint provider).

**Friction:**
- **[Blocker]** Switching verticals **silently discards all active filters** (`verticalSlice.ts:11`). A user who has narrowed by Author and File Type, then taps "Documents" to see only documents, loses both filters with no warning, no confirmation, no undo. (See UX-004 in Appendix A — this is documented as "by design" but still surfaces here as a Day 1 user blocker.) → owning track: T2
- **[Confusion]** Empty verticals (`count === 0`) render as "dimmed" (`VerticalTab.tsx:53-55, isDimmed={isEmpty}`), but `handleClick` ignores clicks on dimmed tabs (`:35`) — there's no tooltip, alert, or status message explaining "this vertical has no results for your current query." The user sees a clickable-looking tab that won't respond. → owning track: T1
- **[Confusion]** Keyboard navigation across verticals is broken. `VerticalTab.tsx:95` sets `tabIndex={isActive ? 0 : -1}` (correct WAI-ARIA roving tabindex), but neither `SpSearchVerticals.tsx` nor `VerticalTab.tsx` implements arrow-key handling to move focus between tabs (only Enter/Space at `VerticalTab.tsx:40-47` activates the focused tab). Tab key cannot reach inactive verticals; arrow keys do nothing. → owning track: Foundations (a11y) / T1
- **[Polish]** When verticals overflow on narrow widths, the OverflowSet "More" menu (`SpSearchVerticals.tsx:201-213`) collapses tabs into a generic Fluent context menu and loses badge prominence (counts appear inline as parentheses at `:189`). On a phone you can't tell which vertical has the most results without opening the menu. → owning track: T1
- **[Polish]** No sort persistence across vertical switches. Sort is preserved in store state (`verticalSlice.ts:11` resets only filters and page, not sort), but if the new vertical's `sortableProperties` doesn't include the previous sort property, `ResultToolbar` silently rejects the sort dropdown choice. There's no toast or message telling the user "your sort no longer applies." → owning track: T2

#### Step 6 — Applies filters

The user clicks a checkbox in the Filters web part — say "FileType: pdf". `handleToggleRefiner` (`SpSearchFilters.tsx:320-347`) computes the next filter set, calls `clearDependentFilters` for filter-config dependencies, then either dispatches `store.setState({ activeFilters, currentPage: 1 })` in `'instant'` mode or stages it into `pendingFilters` in `'manual'` mode. In manual mode, an "Apply filters" bar appears (`:512-523`).

**Friction:**
- **[Confusion]** Manual apply mode shows the "Apply filters" button but no count of pending changes (e.g., "Apply 3 pending changes"). See INC-001 Changed-Form in Appendix A. The user has to remember what they toggled. → owning track: T1
- **[Confusion]** Date range filter (`DateRangeFilter.tsx:27-51`) ships with seven presets (Today / 7 days / 30 days / This month / 3 months / This year / Custom), but uses local-time math for "today" (`:96-103, getStartOfDayUTC(now)`) where the conversion to UTC can place the boundary in the wrong calendar day for users west of UTC after late evening. SharePoint indexes in UTC, so a Pacific-time user clicking "Today" at 6pm will see results from a 31-hour window. → owning track: Foundations
- **[Confusion]** Two parallel apply paths exist — checkbox / dropdown / etc. dispatch via `handleToggleRefiner` (`SpSearchFilters.tsx:320-347`); the Visual Filter Builder dispatches via `handleBuilderApply` (`:350-361`) which calls `clearAllFilters()` then `setRefiner` per filter, bypassing the dependent-filter cascade in `clearDependentFilters`. A user who applies via the builder loses dependent-filter resets that the same selection via checkboxes would have triggered. → owning track: T2
- **[Confusion]** "Show more" toggle exists for Checkbox / TagBox / Dropdown filters (`CheckboxFilter.tsx:209-219`, see MISS-004 Changed-Form in Appendix A) but is **absent** from Taxonomy and People filters. Long taxonomy hierarchies and large people lists rely on type-ahead only — no expandable list view. → owning track: T1
- **[Polish]** Active filter pills (`SpSearchResults.tsx:801-806`) live above the results while the source filter UI is in a separate side web part. Removing a pill via `removeRefiner` doesn't visibly uncheck the matching filter when scrolled out of view, with no scroll-to or highlight animation. → owning track: T1
- **[Polish]** No "AND/OR" indicator when multiple values are selected within one filter. `operatorBetweenFilters` is invisible and per MISS-002 Still-Open in Appendix A may not be fully functional. The user's mental model of "FileType: pdf, docx" is ambiguous. → owning track: T2

#### Step 7 — Switches layouts

The user clicks a layout icon in the Result Toolbar — say List → Card. `ResultToolbar` (`SpSearchResults.tsx:786-797`) renders only layouts in `availableLayouts`, calls `handleLayoutChange` (`:513-515`) which dispatches `setLayout`. Layout chunks are code-split and lazy-loaded; hover preloads via `LAYOUT_PRELOADERS` (`SpSearchResults.tsx:41-47`, `ResultToolbar.tsx:135`).

**Friction:**
- **[Confusion]** A vertical with a configured `defaultLayout` will silently override the user's manual layout choice on the next vertical switch (`SearchOrchestrator.ts:117-133`, `setTimeout((): void => { this._store.getState().setLayout(targetLayout); }, 0)`). The user picks Card, switches to People vertical (which auto-defaults to People layout), comes back to Documents, and finds themselves on List again. The setTimeout-based switch also produces a one-frame flicker (BUG-010 Still-Open WONTFIX in Appendix A). → owning track: T1
- **[Confusion]** Layout switching does not preserve scroll position. After the lazy chunk loads and renders, the user is dropped at the top of the results regardless of which row they were viewing. Pagination's `handlePageChange` does scroll to top intentionally (`SpSearchResults.tsx:531-533`), but layout changes do not preserve the equivalent. → owning track: T1
- **[Confusion]** Sort dropdown is hidden when `activeLayoutKey` is not `list`, `compact`, or `grid` (`SpSearchResults.tsx:793`). Switching to Card / People / Gallery silently removes the sort UI; the previously-applied sort still affects the result order under the hood. The user has no indication that the sort still applies but cannot be changed. → owning track: T1
- **[Polish]** The DataGrid layout has six advertised columns by admin config (`SpSearchResultsWebPart.ts:984-996`), but bulk actions / row selection are NOT wired in (verified at `DataGridContent.tsx` — `grep -ni 'selection' DataGridContent.tsx` returns no Selection import/use; `BulkActionsToolbar.tsx` is defined at 462 lines but **never imported anywhere** — `grep -rn 'BulkActionsToolbar' src | grep -v BulkActionsToolbar.tsx` returns zero hits). `docs/admin-guide.md:140` and `CLAUDE.md:35,162,282` advertise "row selection and bulk actions" — the end user reading the admin's hand-off email expecting to multi-select for share/download will find no checkbox column and no toolbar. → owning track: T2
- **[Polish]** Layout switcher icons in `ResultToolbar` are Fluent monochrome glyphs with no labelled tooltip describing what each layout *looks like*. A user who has never seen "compact" doesn't know whether to click it. → owning track: T1

#### Step 8 — Opens detail panel

The user clicks a result thumbnail or a layout-specific preview affordance. `enablePreviewPanel` is wired through `handlePreviewItem` (`SpSearchResults.tsx:542-544`) which sets `previewPanel.isOpen` in the store, lazy-loading `ResultDetailPanel` (`:830-837`). Inside, a Fluent `Panel` of `PanelType.medium` opens from the right (`ResultDetailPanel.tsx:209-217`) with header / preview / metadata / form-link sections.

**Friction:**
- **[Confusion]** Detail panel has no Close button in the standard Fluent location — `hasCloseButton={false}` is set explicitly (`ResultDetailPanel.tsx:215`) and replaced by `onRenderNavigationContent` which uses an Fluent `IconButton` with `iconName="ChromeClose"` in the navigation slot. The button is small and the icon name is non-obvious; users accustomed to Fluent panels look for the standard X. → owning track: T1
- **[Confusion]** Files that don't support WOPI preview (`supportsWopiPreview` allow-list at `ResultDetailPanel.tsx:48-56` — only Office docs, PDF, OneNote, Visio, txt, csv) fall through to "Preview is not available for this file type" with an "Open in browser" button (`:289-298`). For very common types (.eml, .msg, .png, .jpg, .mp4, .zip) this is the entire panel content — the user gets a label, an icon, and a button, with metadata and history sections still rendering below. The page feels broken. → owning track: T1
- **[Confusion]** WOPI preview takes 2–8 seconds first-time and shows only a 300-px-tall shimmer bar with no progress indication (`ResultDetailPanel.tsx:264-272, 282`). Users who think "it broke" close and re-click, restarting the load. → owning track: T1
- **[Confusion]** "View item" / "Edit item" / "View history" links (`ResultDetailPanel.tsx:381-418`) open in a new WopiFrame window. For an item where the user has no edit permission, the Edit form returns an error page; the panel gives no permission hint up-front. → owning track: T2
- **[Polish]** Author block falls back to `Icon iconName="Contact"` and literal "Unknown" (`ResultDetailPanel.tsx:313-317`) when `item.author?.email` is empty. For pages indexed without author claims this is the most common state and reads as data corruption. → owning track: T1
- **[Polish]** No "next / previous result" navigation in the panel — users who open one result and want the next must close, click, and re-open. PnP Modern Search v4 supports arrow-key paging through results. → owning track: T2

#### Step 9 — Saves the search

The user wants to keep this query+filter combination. Save is gated on the Search Manager being present. The default Search Box ships `enableSearchManager: true` (`SpSearchBoxWebPart.manifest.json:41`), so a `SearchBookmark` icon button appears at the right of the input bar (`SpSearchBox.tsx:809-820`). Clicking opens a Fluent `Panel` (`:879-910`) hosting a `UserSearchManager` instance. The user clicks the "Save" button in the panel header (`SpSearchManager.tsx:716-728`), filling out a Title + Category in a confirm dialog (`:872-967`).

**Friction:**
- **[Confusion]** The Search Box's Save entry point is an unlabelled `IconButton` with `iconName="SearchBookmark"` and `title="My searches"` (`SpSearchBox.tsx:811-818`). The icon glyph reads as "bookmark" / "favorites", not "save current search." The user has to hover for the tooltip and then translate "My searches" into "click here to save the current one." → owning track: T2
- **[Confusion]** Inside the panel, the "Save" header button is disabled (`disabled={!canSave}`, `SpSearchManager.tsx:722`) when `canSave === false` — which evaluates to false when `normalizedQueryText.length === 0 && activeFilters.length === 0` (`:665-666`). On the empty-state page (Step 1), a user who opens the panel sees a greyed-out Save button with no tooltip explanation of why. The TooltipHost wraps the button (`:717`) but only renders "Save the current query and filters" — not "you don't have a query yet." → owning track: T2
- **[Confusion]** When Search Manager is hosted in a *separate* web part (per `SpSearchManagerWebPart`), AND Search Box also enables the inline manager, the user sees two entry points to the same set of saved searches. (See Journey A Step 6 [Polish].) → owning track: T2
- **[Polish]** The Save dialog "What will be saved" preview (`SpSearchManager.tsx:914-953`) shows query, filters, vertical, and sort — but not the scope, layout, or pagination state that are also persisted (`:538-545`). Users see a partial inventory. → owning track: T2
- **[Polish]** Saved-search JSON parses without schema validation on restore (see SEC-004 Still-Open in Appendix A). A tampered shared search can poison the store with no end-user error UI. → owning track: Foundations

#### Step 10 — Shares the search with a colleague

The user clicks Share on either the inline Save panel header (`SpSearchManager.tsx:730-743`) or on a row in the Saved Searches tab (per-row Share action passed via `onShare`). `ShareSearchDialog.tsx` opens (`:206-380`) with four tabs: Copy Link / Email / Teams / Users.

**Friction:**
- **[Blocker]** "Share to Users" calls `service.shareToUsers` (`ShareSearchDialog.tsx:174-204`, `SearchManagerService.ts:1349-1408`), which sets `SharedWithId`, breaks role inheritance, and grants read permission — but **sends no notification of any kind to the recipient**. No email, no Teams chat, no Activity Feed entry. The recipient has the search shared with them and never knows it exists until they happen to open Search Manager and check the Shared Searches tab. The dialog success message says "Search shared successfully" (`:306`), which the sharer will reasonably interpret as "they got a notification." → owning track: T2
- **[Confusion]** Email tab opens a `mailto:` link with the share URL in the body (`ShareSearchDialog.tsx:130-147`); recipients on a different SharePoint site will be redirected or 403'd on click, with no tenant/site-context warning. → owning track: T2
- **[Confusion]** Teams tab opens `getTeamsBaseUrl() + '/l/chat/0/0?message=…'` (`ShareSearchDialog.tsx:150-159`). The message is a *draft* in the Teams web client, not a sent chat — but the dialog says "Teams chat opened" (`:280`), suggesting it was sent. → owning track: T2
- **[Confusion]** "Share to Users" requires `search.id > 0` (`ShareSearchDialog.tsx:42`). When initiated for an *unsaved* current search, the dialog falls through to placeholder text "People-based sharing requires a saved search with SharePoint sharing enabled" (`:361-367`). The user has to back out, save, then re-open Share. → owning track: T2
- **[Polish]** Copy Link tab shows the URL in a read-only `TextField` (`ShareSearchDialog.tsx:229-244`). When the URL is the `?sid=` fallback (`urlSyncMiddleware.ts:601-623`), the link is opaque — `…/Search.aspx?sid=42` — and the user has no idea what it encodes. → owning track: T2
- **[Polish]** Sharing to multiple users iterates `ensureUser` + `roleAssignments.add` sequentially (`SearchManagerService.ts:1369-1402`). For 10 recipients this takes 5–10 seconds with only a `Spinner label="Sharing..."` (`ShareSearchDialog.tsx:349`) and no per-user progress count. → owning track: T2

#### Step 11 — Returns later via deep link

The user (or a colleague) opens a previously-shared URL: `https://…/Search.aspx?q=annual+report&v=documents&ft=pdf&s=LastModifiedTime:desc&p=2&l=card`. `createUrlSyncSubscription` (`urlSyncMiddleware.ts:665-783`) deserializes on init (`:675`), applies state to the store (`:697-712`), and starts a 5-second `pendingFilterTimeout` to abandon URL filters if `filterConfig` doesn't arrive in time (`:701-712`, see BUG-003 closed in Appendix A).

**Friction:**
- **[Blocker]** All URL writes use `window.history.replaceState` (`urlSyncMiddleware.ts:617, 622, 625, 862`) — never `pushState` — even though the function is named `pushStateToUrl` (`:578`). Consequence: the browser **Back button does not move through search history**. A user who runs three queries, then clicks Back to revisit the second, leaves the SP Search page entirely. This breaks the most basic web navigation contract and is invisible until the user tries it. → owning track: T2
- **[Confusion]** On URL load with `?l=card` requesting a layout NOT in the admin-configured `availableLayouts`, `applyUrlStateToStore` coerces the layout to the first available and rewrites the URL via `replaceState` (`urlSyncMiddleware.ts:837-863`, see BUG-008 closed in Appendix A). The user receives a different page than the link they clicked; no toast, no message bar tells them "we changed your view because Card layout isn't enabled here." → owning track: T3
- **[Confusion]** When the URL exceeds 2,000 chars, it is replaced with `?sid=<id>` pointing to a `SearchSavedQueries` row (`urlSyncMiddleware.ts:601-623`). If the row was deleted by 90-day retention cleanup (`SearchManagerService.ts:735-739, 1417`), the URL silently fails to restore — user lands on the default view with no "link expired" notice. → owning track: T2
- **[Confusion]** URL filter aliases default to short codes — `ft`, `au`, `md`, etc. (`filterUrlAliases.ts:18-46`). When two managed properties normalize to the same alias (e.g. `Author` and `AuthorOWSUSER` both producing `au`), the second silently overwrites the first; no collision detection in `getDefaultFilterUrlAlias` (`:12-47`). → owning track: T3
- **[Polish]** If Filters web part registers `filterConfig` AFTER the orchestrator's first search, URL filters apply to a second search 100–500 ms later, producing a flash of "all results" then "filtered results" on deep-link first paint. → owning track: T3
- **[Polish]** The 5-second `URL_FILTER_RESTORE_TIMEOUT_MS` (`urlSyncMiddleware.ts:54`) only writes `console.warn` (`:705-708`). Users on a slow tenant silently lose URL filters; only a developer with dev tools open will see the warning. → owning track: T5

#### Step 12 — Repeats on mobile

The user opens the same page on their phone. The five web parts stack vertically by SharePoint page layout. Mobile responsiveness is implemented per-component via SCSS media queries — the breakpoint set varies wildly by web part.

**Friction:**
- **[Blocker]** The Filters web part has **zero `@media` queries** (`grep -c '@media' src/webparts/spSearchFilters/components/SpSearchFilters.module.scss` returns `0`). On a phone-width viewport, the filter panel is rendered at its desktop width with no responsive collapse, no off-canvas behaviour, and no "show filters" toggle — it stacks above the results consuming the full screen height. The user has to scroll past the entire filter list to reach the results. → owning track: T1
- **[Confusion]** Mobile breakpoints are inconsistent across web parts: Search Box uses `480px` (`SpSearchBox.module.scss:774`); Verticals uses `640px` (`SpSearchVerticals.module.scss:195`); Results mixes `1023px / 639px / 399px` (`SpSearchResults.module.scss:1338, 1342, 1393, 2303, 2307, 2311, 2444`); Manager uses `480px / 640px / 768px / 900px` (`SpSearchManager.module.scss:116, 945, 1016, 1050, 1126, 1407, 1467, 1519`). At 481 px Search Box is desktop-mode but Manager has already collapsed action labels. → owning track: T1
- **[Confusion]** Verticals overflow detection (`SpSearchVerticals.tsx:51-107`) measures pixel width but the OverflowSet "More" menu opens as a Fluent `ContextualMenu` styled for desktop hover. Tap targets are small and the menu can render off the right edge of the viewport on phones. → owning track: T1
- **[Polish]** Detail panel (`PanelType.medium`, `ResultDetailPanel.tsx:212`) collapses to full-width on mobile, but its inner sections have no SCSS rules narrower than the desktop layout (`SpSearchResults.module.scss` Panel blocks). Action bar buttons wrap awkwardly; preview iframe forces horizontal scroll. → owning track: T1
- **[Polish]** Search Box at `max-width: 480px` (`SpSearchBox.module.scss:774-790`) keeps the KQL toggle pills, Search Manager, and Query Builder buttons all inline (`SpSearchBox.tsx:735-762, 809-832`), pushing the input field to ~140 px wide. Typing more than a short query is uncomfortable. → owning track: T1
- **[Polish]** Fluent `SearchBox` font size is 14 px; mobile browsers zoom in on focused `<input>` with `font-size < 16px` and don't zoom back out on blur. → owning track: T1

## Part 2 — Differentiator Tracks

### T1. Modern UI Quality

#### Current state

SP Search ships visual surfaces across six web parts, all built on Fluent UI v8 with theme-token CSS variables wired through `*.module.scss` files. Theming integration is consistent at the token level — every web part uses `"[theme:tokenName, default: #hex]"` plus `var(--tokenName, fallback)` pairs (e.g. `SpSearchResults.module.scss:7-8,31-32,43-44,49-50,67`), so SharePoint section themes propagate without code changes. Loading states use Fluent `Shimmer`. Active-filter pills, layout cards, and the detail-panel overlay carry hand-tuned `transition` (150–200ms ease) and a small set of `@keyframes` (`pillFadeIn`, `overlayFadeIn` at `SpSearchResults.module.scss:197,206,1132,1135`). #### Gap to "amazing"

The audience profile is "any SPFx tenant, self-serve, no author hand-holding" — the end user who lands on a published search page must feel that saving, sharing, and re-finding work the way every other modern web tool teaches them to expect. A self-serve tenant evaluating SP Search compares it to Confluence search, Notion search, GitHub search — all of which ship at least three of those four affordances. The architecture and the data layer are right; the surface area between the user and the data is the gap.

### T3. Multi-Instance / Multi-Context

#### Current state

T3 is the project's strongest differentiator versus PnP Modern Search v4 — PnP has no multi-context model at all (Appendix C "Multi-instance isolation via `searchContextId`" — "PnP v4 uses Dynamic Data which couples instances"). The implementation rests on three deliberate architectural choices: a window-backed singleton context map that survives webpack entry-point duplication, a stable per-context URL prefix computed at context creation, and a per-vertical `dataProviderId` field that lets one context route different verticals to different providers. The substrate is real, but the surface area exposed to admins and end users is thin in three places: collision detection, lifecycle cleanup, and URL alias namespacing.

The actual T3 surface, by file:

- **`searchContextId` correctness.** Every web part declares `searchContextId: string` with manifest default `"default"` (`SpSearchBoxWebPart.manifest.json:25`, `SpSearchResultsWebPart.manifest.json:25`, `SpSearchFiltersWebPart.manifest.json:25`, `SpSearchVerticalsWebPart.manifest.json:25`, `SpSearchManagerWebPart.manifest.json:25`, `SpSearchAdminManagerWebPart.manifest.json:18`). Each web part normalizes via `this.properties.searchContextId || 'default'` (e.g. `SpSearchBoxWebPart.ts:129`, `SpSearchResultsWebPart.ts:153,189`, `SpSearchVerticalsWebPart.ts:91`, `SpSearchFiltersWebPart.ts:121`, `SpSearchManagerWebPart.ts:139`). All web parts re-resolve the store when the property changes (`SpSearchResultsWebPart.ts:852-856`, `SpSearchVerticalsWebPart.ts:200-204`, `SpSearchFiltersWebPart.ts:234-237`).
- **URL parameter namespacing.** `urlSyncMiddleware.ts:79-81` defines `prefixKey(key, prefix)` which produces `{prefix}.{key}` when prefix is set. Reserved param keys (`q`, `v`, `s`, `p`, `l`, `x`, `i`) and filter aliases all flow through this path (`urlSyncMiddleware.ts:87-95, 97-99, 105-113`). The prefix itself is computed deterministically at context creation via `_buildStableUrlPrefix(searchContextId)` (`storeRegistry.ts:217, 292-306`) — a normalized 6-char stem + 6-char FNV-1a hex hash, stable across reloads. `pushStateToUrl` debounce timers are keyed per-prefix (`urlSyncMiddleware.ts:586-593`), so two contexts cannot stomp each other's pending URL writes.
- **Per-vertical `dataProviderId` routing.** `IVerticalDefinition.ts:16` declares `dataProviderId?: string`. `SearchOrchestrator._getProvider(state)` (`SearchOrchestrator.ts:544-557`) reads the active vertical and prefers a provider matching `vertical.dataProviderId`, falling back to the first registered provider. `Provision-TestPages.ps1:354-380` exercises this path with the `multi-people` context routing the People vertical to Graph while Documents stays on SharePoint Search.
- **Cross-context coordination.** Each context owns a separate `ISearchContext` (`storeRegistry.ts:18-26`) with its own store, orchestrator, registries, and URL sync subscription. There is no cross-context publish/subscribe API; coordination is intentionally absent so isolation is the default contract.
- **Library-component singleton backing.** `storeRegistry.ts:36-54` stores both `__sp_search_context_map__` and `__sp_search_init_promises__` on `window` to defeat webpack entry-point duplication. Without this, every web part bundle would receive its own module-scoped Map and "shared" contexts would in fact be N independent stores — the failure mode SP Search exists to prevent. The init-promises map (`storeRegistry.ts:119-138`) serializes concurrent `initializeSearchContext()` calls so two web parts racing to bootstrap the same context resolve once.
- **Multi-context URL prefix race repair.** `storeRegistry.ts:224-234` re-subscribes ALL previously-initialized contexts when `map.size` transitions from 1 to 2, so the first-loaded context picks up its stable prefix mid-session (BUG-005 Closed in Appendix A; `storeRegistry.ts:217` computes the prefix at context creation rather than at first URL write).
- **Sample multi-context pages.** `scripts/Provision-TestPages.ps1:319-381` provisions `test-multi-context` with two independent contexts (`multi-docs`, `multi-people`) — Search Box + Results + Filters per context, scenario-preset-tuned, on one page. No documentation page exists today (`docs/admin-guide.md` mentions multi-context only in passing at `:3` and lists `searchContextId` in the per-web-part property tables; `docs/extensibility-guide.md:9, 537-552` documents the registry pattern but ships no walkthrough or screenshots).
- **Edit-mode default-context advisory.** `SpSearchResults.tsx:716-720` renders an edit-only `MessageBar` when `searchContextId === 'default'`. This is the only in-product hint that multi-context awareness exists. There is **no** detection or warning for the inverse case (Box uses `default`, Results uses `hr-search` — orphaned web parts). There is no edit-mode warning for any of the other five web parts.
- **Property pane discoverability.** The `searchContextId` field lives at inconsistent locations: page 3 of 4 in Search Box (`SpSearchBoxWebPart.ts:362-375`), page 3 of 4 in Search Results (`SpSearchResultsWebPart.ts:1207-1228`), page 2 of 3 in Search Filters (`SpSearchFiltersWebPart.ts:443-463`), **page 1 group 1** in Verticals (`SpSearchVerticalsWebPart.ts:317-330`), and page 1 group 1 in Manager (`SpSearchManagerWebPart.ts:201-215`). Three distinct description strings ship across the five web parts (`spSearchBox/loc/en-us.js:17`, `spSearchResults/loc/en-us.js:17`, `spSearchFilters/loc/en-us.js:14`).
- **Filter URL alias collision risk.** `filterUrlAliases.ts:18-46` maps short codes (`au`, `md`, `ft`, `ct`, `ti`, `pa`, `sz`, `cr`, `cc`) without uniqueness checks. #### Gap to "amazing"

The audience profile is "any SPFx-capable tenant, self-serve, no author hand-holding" — and T3 is the moat. PnP Modern Search v4 has no multi-context concept at all, so a tenant evaluating SP Search alongside PnP needs to see the moat without reading the source. Today four failure modes erode that promise. **First, collision detection is silent in both directions**: the only edit-mode advisory fires when `searchContextId === 'default'` on the Results web part — the inverse case (Results uses `hr-search` while Box and Filters still use `default`, or any pairwise mismatch between connected web parts) shows no warning anywhere, and an admin who mistypes one ID gets a search page where the box and results don't talk to each other with no in-product diagnostic. **Second, lifecycle cleanup is a leak in waiting**: `disposeStore` exists, is correct, and is wired to nothing — so a SharePoint SPA shell navigation away from a search page leaves a context, orchestrator subscription, popstate listener, and debounce timer alive on `window.__sp_search_context_map__` indefinitely; over a long session a single tab accumulates orchestrators, each subscribing to its own (now-invisible) store. **Third, URL alias namespacing has a per-context gap**: short aliases (`au`, `md`, `ft`) are computed deterministically without uniqueness checks, so two managed properties in the same filterConfig that normalize to the same alias collide silently — the second filter's value wins on write, the first vanishes from deep links. The collision space gets denser in multi-context pages because each context owns its own alias namespace shape but admins authoring two contexts side-by-side don't see the shared-alias risk in any property pane validation. **Fourth, the moat is undocumented**: a `test-multi-context` provisioning page exists but there is no admin doc, no walkthrough, no screenshot of the resulting URL (`?ctx1.q=...&ctx2.q=...`), no troubleshooting flowchart. A self-serve admin reading the docs cannot find a single sentence that says "to put two independent searches on one page, set searchContextId to two different values." The architecture protects them; the docs do not advertise it.

#### Out of scope for v1.0

- **Cross-context publish/subscribe API.** Rationale: Spec §4.3 T3 lists "cross-context coordination patterns" as scope but the architecture intentionally ships isolation as the only contract — adding a coordination API risks turning multi-context pages into a coupled-by-default model that re-creates the PnP Dynamic Data failure mode SP Search exists to avoid. Capture as v2.0 only if user research shows real coordinate-two-contexts demand (e.g. "one search drives a related search").
- **`searchContextId` change live-migration of in-flight state.** Rationale: When the admin changes `searchContextId` from `default` to `hr-search` in property pane, the existing default store keeps its state and a fresh `hr-search` store starts empty. A live-migration would copy state across the boundary atomically. This is rare in practice (admins set the ID once at page-build time, not at runtime) and would re-introduce the cross-context coupling we explicitly reject in Out of Scope item 1. Defer.
- **Per-context CSP nonce and iframe sandbox attribution.** Rationale: SP Search inherits the SharePoint page CSP; per-context security boundaries (e.g. one context can't read another's preview iframe) would require shadow DOM + per-context iframe rewrites. Out of audience-profile scope ("self-serve any tenant") — different product shape. Capture in v2.0 if a multi-tenant or guest-context use case materializes.
- **Cross-page state propagation via tenant-shared backend.** Rationale: Same `searchContextId` on two SharePoint pages currently produces two independent stores (window-backed map is per-tab) — by design, since cross-tab/cross-page sync would require either BroadcastChannel (no SharePoint guarantee) or a tenant-shared list write per state change (perf disaster). Documented as expected behavior in D5; not a deliverable.
- **Telemetry on multi-context usage (how many tenants run two contexts vs one).** Rationale: Out of audience-profile scope at v1.0 (Foundations track owns the broader telemetry decision per spec §4.4); usage telemetry would need to land alongside the tenant-wide opt-in Foundations work, not as a T3-specific gate.
- **Programmatic enforcement of init order (e.g. block `initializeSearchContext` until all expected web parts have registered).** Rationale: ARCH-003 deliverable D10 ships a diagnostic warning instead of a hard enforcement; a hard block would create new failure modes (e.g. a page where the admin removed Filters web part would never finish init). Diagnostic is the correct v1.0 shape; enforcement is post-v1.0 if data shows ongoing init-order pain.
- **Window-key namespace consolidation under a single `window.__sp_search__` object.** The existing `__sp_search_*` prefix is intentionally chosen as the SP Search namespace; forks or unrelated solutions using the same prefix will silently merge. Consolidating the three (and any future) globals — `__sp_search_context_map__`, `__sp_search_init_promises__`, `__sp_search_webpart_registrations__` (introduced by D2) — under one object is deferred to v1.1+ — no observed conflict in the wild and the 1-time migration cost is moderate.

#### Gap to "amazing"

The audience profile is "any SPFx-capable tenant, self-serve, no SI hand-holding" — and T4 is the surface where the admin decides in the first 30 minutes whether SP Search is a configurable product or a custom build. Today the gap is concentrated in five places, each independently shippable. **First, the provisioning scripts are unsafe by default**: `Setup-SPSearchSite.ps1` silently destroys any page named `Search` on every re-run with no `-WhatIf`, no confirm prompt, no `-Force` opt-in — the admin who tweaks the auto-provisioned page in the SharePoint UI and then re-runs the script (because they want to add a second context, or fix a typo, or roll back a partial install) loses everything. Pair that with seeded coverage profiles that point at hardcoded test-data URLs (`CorporatePolicies`, `SalesMaterials`, …) that don't exist on a clean tenant, and the very first thing the admin sees post-publish is the Admin Dashboard rendering broken stat cards. **Second, the preset picker is buried and stops at the Results web part boundary**: nine presets ship (correcting the stale "Sprint 4 backlog" note in CLAUDE.md), but the `PropertyPaneChoiceGroup` lives on page 2 of the Results pane inside the layouts group, the picker omits `account-documents` entirely, and selecting `documents` does not push the preset's `filterSuggestions` or vertical hints into the Filters / Verticals web parts — so the admin who picks a preset still has to hand-key the filter and vertical setup that the preset was supposed to deliver. The "preset" promise is half-kept. **Third, the schema picker is wired to one field**: `PropertyPaneSchemaHelper` is a real product surface that browses tenant managed properties with refinable/sortable/retrievable/queryable filters — but it only assists `queryTemplate`. Every column picker, every filter `managedProperty` field, every sort property, every collapse spec, and every per-vertical `dataProviderId` falls back to free-text. A typo in `LastModifedTime` produces a silently empty column at runtime with no diagnostic anywhere. **Fourth, the Admin Dashboard's first paint is fragile**: the four panels are wired and look polished when populated, but Coverage opens to broken cards on day 1, Quality / Insights have no per-user or per-vertical breakdown (UX-006 Still-Open), the zero-results panel needs manual refresh (UX-005 Changed-Form), and CTR is a single window-wide number with no week-over-week trend (UX-007 Still-Open). The dashboard feels like four monitors taped to a wall instead of a curated admin surface. **Fifth, Admin Manager differentiation is missing**: two manifests ship two near-identical toolbox tiles, both titled "Manager", both icon `SearchAndApps`, one labelled `(Legacy)` — and the only behavioral difference is one `DebugCollector` call. INC-007 closure left the architectural gap intact. The architecture is mostly right; what is missing is the polish that turns "configurable" into "configurable in 15 minutes by an admin who has never seen this product."

#### Positioning takeaways

- **SP Search wins on differentiators that PnP v4 doesn't even attempt.** Search Manager (saved/shared/collections/history), Admin Dashboard (Health + Insights), Debug FAB, multi-context isolation via `searchContextId`, scenario presets, KQL editor, and DataGrid layout are all entirely-missing-in-PnP capabilities that anchor T2/T3/T4/T5 differentiators.
- **Filter and refiner catalogue is at parity with one notable gap and one notable strength.** SP Search ships Slider/Text/Toggle and a Visual Filter Builder that PnP doesn't, but lacks PnP's Date Interval (relative-range buckets) — a T1/T4 polish item, easy add.
- **Data source breadth is a real positioning weakness.** PnP v4 ships SharePoint + Microsoft Search + Azure AI Search + custom; SP Search ships SharePoint + Graph only. Per-vertical `dataProviderId` routing softens the gap, but Azure AI Search support is increasingly table stakes for enterprise demos. Decide explicitly whether to invest (T4 deliverable) or document as out-of-scope (Appendix D).
- **Templating paradigm is a deliberate divergence, not a gap to close.** PnP's Handlebars + web components is genuinely more flexible for non-developer admins; SP Search's React + cell renderers + registries is more type-safe and tree-shakable. Document this as a positioning choice in launch materials so prospects don't expect Handlebars.
- **Cross-cutting hygiene (accessibility conformance statement, audience targeting, telemetry) is the area where SP Search is materially behind.** None of these tie to a stated differentiator on their own, but each one fails the "self-serve any tenant" launch bar (spec §3.1) in ways an enterprise procurement review will catch. Foundations track.
